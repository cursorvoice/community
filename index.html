How plugins work

Cursor Voice loads every .json manifest in:

~/Library/Application Support/CursorVoice/plugins/

Each manifest defines one tool: a name, a description (so the model knows when to use it), its parameters, and a run action — a shell command, applescript, or open_url template with {{argument}} placeholders.

Manifest format

{
  "name": "search wikipedia",
  "description": "Open a Wikipedia search for a topic in the browser.",
  "parameters": {
    "type": "object",
    "properties": { "topic": { "type": "string" } },
    "required": ["topic"]
  },
  "run": {
    "type": "open_url",
    "template": "https://en.wikipedia.org/w/index.php?search={{topic}}"
  }
}

Then just say “search Wikipedia for the Eiffel Tower.” The tool name is exposed to the assistant as plugin_search_wikipedia. Run types:

  • open_url — opens a URL (args are URL-encoded).
  • shell — runs a zsh command (args are shell-quoted; gated by the destructive-command guard and Dry-run).
  • applescript — runs AppleScript (args are escaped).

⚠️ Plugins run with your permissions. Only install manifests you trust — review the run template before adding one.

Submit your plugin

Share a plugin with everyone:

  1. Fork cursorvoice/community.
  2. Add your manifest to plugins/your-plugin.json.
  3. Add an entry to registry.json (name, file, type, description, author).
  4. Open a pull request. Once merged, it shows up here.
